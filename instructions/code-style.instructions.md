# [代码风格与验证规则]

## 注释规则

1. 实际业务功能位置必须提供中英日三语注释或说明。
2. 优先覆盖这些位置：Wails 绑定方法、核心 Worker、网关管理、任务调度、关键页面逻辑、关键 SQL。
3. 注释要解释业务目的、调用链、边界条件和失败时行为，不要只重复代码字面含义。

## 代码风格

1. 保持直接、具体、少废话。
2. 先把数据流、状态流转和错误路径写清楚，再考虑抽象层次。
3. 避免为了“优雅”引入和当前仓库不匹配的大而全封装。

## 最低验证标准

1. 代码可运行或可构建。
2. 有最小可执行验证路径。
3. 桌面功能改动后，至少说明如何通过 wails dev 和对应页面进行人工验证。
4. 数据库或 SQL 改动后，至少给出查库或复查路径。
5. 涉及高频写入表（sys_task_execution_logs、sys_data_history）的改动，必须评估写入频率是否会在生产环境导致磁盘满。
6. 涉及历史修复 SQL 时，默认写成可预览、可回滚脚本：先备份/事务、再 SELECT 差异、UPDATE、复查，最后默认 ROLLBACK，确认后才改 COMMIT。
7. 工单数量修复脚本必须写清口径：actual_qty 是总产量增量，ng_qty 是 NG 脉冲净值，ok_qty = actual_qty - ng_qty。

## 高频写入风险检查

高频写入场景必须满足以下至少一项：
- 有写入频率上限（如冷却时间、边沿触发而非电平触发）
- 有数据保留策略（定期清理旧数据）
- 有磁盘占用监控（表 data_MB 异常时能被发现）

**反例（已出现的故障）**：条件任务（task_type=3）改为"条件为真即触发"，设备持续离线时每5秒写一条日志，数小时内积累200万行，磁盘100%导致应用卡死。
# [代码风格与注释指令]

## 注释语言要求
- 实际功能位置必须提供中英日三语注释或说明。
- “实际功能位置”包括但不限于：Controller、Service、Entity、Repository 中的特殊查询、核心页面逻辑、关键状态流、关键 SQL、推荐算法段、缓存键设计。
- 简单 getter/setter、显而易见的样板代码，不要求强行塞三语注释。

## 注释内容要求
- 注释要解释为什么要这么做，而不是重复代码表面意思。
- 优先写清以下信息：业务目的、输入输出、边界条件、失败时行为、降级逻辑、与其他服务或数据库的关系。
- 三语注释应保持语义一致，不要三种语言各写各的。

## 代码风格要求
- 默认采用 Linus Benedict Torvalds 风格。
- 规则是：直接、具体、少废话、优先可读性、反对无意义抽象和花哨包装。
- 能用一层解决的，不要先上三层抽象。
- 能明确命名的，不要依赖隐晦缩写。
- 能把错误路径写清楚的，不要把风险藏进“优雅”封装里。

## 执行优先级
1. 先保证业务逻辑正确。
2. 再保证代码路径清晰。
3. 最后再考虑是否需要抽象复用。

## 测试要求
- 功能模块必须带测试用例，至少覆盖最小可验证路径。
- 如果当前阶段只先做联调接口，也要保留能让人手动验证的请求路径、样例入参与预期结果。
- 不允许只写“看起来能跑”的代码而没有任何验证入口。
