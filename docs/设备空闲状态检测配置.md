# 设备空闲状态检测配置

## 🎯 **需求说明**

**触发条件**（三个条件同时满足）：
1. **开机状态的质量码 = 1**（设备在线，通信正常）✅
2. **开机状态的值 = 1**（设备开机）✅
3. **停机状态的值 = 1**（设备停机）✅

**逻辑**：设备在线 + 开机ON + 停机ON = **空闲状态**

**动作**：更新设备状态为 `3`（空闲）

---

## 💡 **实现方案**

使用 **数据改变任务 + 条件数据库操作** 实现：
- 监控停机状态变化
- 触发时检查所有条件
- 条件满足才更新设备状态

## 📝 **配置说明**

### **action_config 字段解析**

```json
{
  "operation": "update_device_status_conditional",  // 🔥 新增的条件更新操作
  "op_params": {
    "device_id": 1,                                 // 设备ID
    "status": 3,                                    // 目标状态（3=空闲）
    "remark": "#1设备进入空闲状态",                  // 备注
    "conditions": {                                 // 🔥 条件配置
      "开机状态_var_id": 76,                         // 开机状态变量ID
      "开机状态_value": 1,                           // 期望值：开机=1
      "开机状态_quality": 1,                         // 期望质量码：在线=1
      "停机状态_var_id": 73,                         // 停机状态变量ID
      "停机状态_value": 1                            // 期望值：停机=1
    }
  }
}
```

### **条件配置规则**

每个条件由三部分组成（以 `开机状态` 为例）：

| 键名 | 说明 | 示例值 |
|------|------|--------|
| `开机状态_var_id` | 变量ID | `76` |
| `开机状态_value` | 期望的数据值 | `1` |
| `开机状态_quality` | 期望的质量码 | `1`（在线） |

**前缀名称**（如 `开机状态`、`停机状态`）可以自定义，但必须保持一致。

## 🔍 **工作流程**

### **流程图**

```
MQTT消息到达
  ↓
LogicWorker 更新 #1停机状态
  ↓
触发任务：#1设备空闲状态检测
  ↓
EventProcessor 执行条件检查：
  ├─ 检查 #1开机状态 质量码 = 1? ✓
  ├─ 检查 #1开机状态 值 = 1? ✓
  └─ 检查 #1停机状态 值 = 1? ✓
  ↓
所有条件满足 ✅
  ↓
执行：UpdateDeviceStatus(device_id=1, status=3)
  ↓
设备状态更新为"空闲"
```

### **条件不满足时**

```
MQTT消息到达
  ↓
LogicWorker 更新 #1停机状态
  ↓
触发任务：#1设备空闲状态检测
  ↓
EventProcessor 执行条件检查：
  ├─ 检查 #1开机状态 质量码 = 1? ✓
  ├─ 检查 #1开机状态 值 = 1? ❌ (实际=0)
  └─ 检查 #1停机状态 值 = 1? ✓
  ↓
条件不满足 ❌
  ↓
跳过更新，记录日志
```

## ⚠️ **注意事项**

### **1. 条件检查时机**

- 只有当 **触发变量**（停机状态）变化时才会检查条件
- 如果开机状态变化但停机状态不变，不会触发检查
- 建议同时监控开机状态和停机状态（创建两个任务）

### **2. 条件检查是实时的**

条件检查时会读取内存中的最新值，而不是数据库中的历史值。

### **3. 质量码可选**

如果不需要检查质量码，可以省略 `_quality` 条件：

```json
"conditions": {
  "开机状态_var_id": 76,
  "开机状态_value": 1,
  "停机状态_var_id": 73,
  "停机状态_value": 1
}
```

### **4. 支持多个条件**

可以添加更多条件，只要遵循 `前缀_var_id`、`前缀_value`、`前缀_quality` 的命名规则：

```json
"conditions": {
  "开机状态_var_id": 76,
  "开机状态_value": 1,
  "开机状态_quality": 1,
  "停机状态_var_id": 73,
  "停机状态_value": 1,
  "温度_var_id": 80,
  "温度_value": 25
}
```
