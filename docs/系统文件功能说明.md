## 🔥 核心工作协程 (workers/)

### 1️⃣ `logic_worker.go` - 逻辑处理核心 ⭐⭐⭐
**职责**: MQTT消息解析 + 内存更新 + 业务分发  
**协程数**: 20个并发  
**输入**: `LogicChan` (MQTT原始消息)  
**输出**: `ChangeChan`, `CycleChan`, `AlarmChan`, `SSEChan`, 触发任务  
**关键逻辑**:
- **Pass 1**: 解析JSON → 更新内存Map → 记录变化
- **Pass 2**: 基于快照执行业务（存储判断、报警检查、任务触发）
- 🎯 **双循环一致性保证**: 所有变量先更新完，再执行业务逻辑
- **采集层防抖**: 数值点位先调用 `models.Tag.ApplyNumericSample()`，确认后才更新内存。产量累计计数器配置 `suspicious_value=0` 时，假 `0` 不会进入历史入库、任务、报警或 SSE。
- **启动快照保护**: 变化存储/混合存储的首帧快照由 `ShouldStartupSnapshot()` 控制；若首帧是防抖可疑值，会先挂起观察，不直接 InitOnly 入库。

**何时修改**: 需要支持新的JSON格式、新的数据类型、新的业务逻辑

---

### 2️⃣ `store_worker.go` - 数据存储执行器
**职责**: 批量写入历史数据到数据库  
**协程数**: 
- `ChangeWorker` x5 (变动存储)
- `CycleWorker` x5 (定时存储)

**输入**: 
- `ChangeChan` (值变化触发)
- `CycleChan` (定时扫描触发)

**批量策略**:
- 变动存储: 每100ms 或 累积10条
- 定时存储: 每200ms 或 累积20条

**何时修改**: 需要优化批量大小、写入频率、存储格式

---

### 3️⃣ `scanner.go` - 定时扫描器
**职责**: 每1秒扫描内存，检查定时存储周期  
**协程数**: 1个单例  
**输出**: `CycleChan`  
**扫描逻辑**:
```go
for 每个Tag {
    if (store_mode=2或3) && (周期到期) {
        发送到 CycleChan
    }
}
```

**⚠️ 当前问题**: 
- 如果变量值为0且从未更新过，会被跳过（见 `models/tag.go:147`）
- **解决方案**: 修改 `ShouldCycleStore()` 逻辑，允许存储0值

**何时修改**: 需要调整扫描频率、定时存储判断逻辑

---

### 4️⃣ `alarm_worker.go` - 报警处理器
**职责**: 写入报警记录 + 更新报警状态  
**协程数**: 3个  
**输入**: `AlarmChan`  
**处理类型**:
- 数值报警: HH/H/L/LL (超限报警)
- 系统报警: SYS (设备故障、错误码)

**何时修改**: 需要添加邮件/短信通知、报警升级逻辑

---

### 5️⃣ `task_scheduler.go` - 任务调度器 ⭐⭐
**职责**: 管理3种任务的触发  
**协程数**: 1个单例  
**任务类型**:
1. **定时任务** (TaskType=1): 每秒检查 `interval_sec`
2. **数据改变任务** (TaskType=2): 由 `LogicWorker` 主动触发
3. **条件事件任务** (TaskType=3): 条件表达式求值 (TODO)

**触发流程**:
```
LogicWorker Pass 2 → TriggerDataChangeTask()
                   ↓
          检查 change_type (ANY/FALSE_TO_TRUE/等)
                   ↓
          非阻塞发送到 EventChan
```

**何时修改**: 需要支持新的变化类型、Cron表达式、条件引擎

---

### 6️⃣ `event_processor.go` - 事件执行器 ⭐⭐
**职责**: 执行任务的5种动作  
**协程数**: 3个并发  
**输入**: `EventChan`  
**动作类型**:
1. **HTTP请求** (1): GET/POST/PUT
2. **MQTT发布** (2): 网关感知、SCADA格式
3. **数据库操作** (3): 原始SQL 或 预定义操作
4. **脚本执行** (4): bash/python/powershell
5. **日志写入** (5): INFO/WARN/ERROR

**模板变量**: 支持 `{{var_id}}`, `{{new_value}}`, `{{old_value}}` 等

**何时修改**: 需要支持新的动作类型、新的预定义数据库操作

## 📦 数据模型 (models/)

### `tag.go` - 测点内存模型 ⭐⭐⭐
**核心结构**: 
```go
type Tag struct {
    VarID, VarName, DataType  // 基础配置
    CurrentValue, LastValue    // 运行时状态
    AlarmEnable, LimitHH/H/L/LL // 报警配置
    StoreMode, StoreCycle      // 存储配置
    mu sync.RWMutex            // 并发锁
}
```

**关键方法**:
- `ApplyNumericSample()`: 数值采样入口，先执行采集层防抖，再返回真实变化
- `UpdateValue()`: 线程安全更新值（兼容旧调用，普通数值更新）
- `ShouldStore()`: 判断变动存储（检查死区）
- `ShouldCycleStore()`: 判断定时存储（检查周期）⚠️ **问题所在**
- `CheckAlarm()`: 检查报警状态
- `ShouldStartupSnapshot()`: 判断冷启动首帧是否只入库不触发任务/报警

**何时修改**: 需要支持新的数据类型、新的存储判断逻辑

---

### `task.go` - 任务配置模型
**定义**: 任务类型、触发条件、动作配置  
**关键字段**:
- `TaskType`: 1=定时, 2=数据改变, 3=条件
- `TriggerVarID`: 触发变量ID
- `ChangeType`: ANY/FALSE_TO_TRUE/THRESHOLD 等
- `ActionType`: 1=HTTP, 2=MQTT, 3=DB, 4=Script, 5=Log
- `ActionConfig`: JSON配置字符串

---

### `message.go` - 通道消息模型
**定义**: 各个Channel传递的消息结构
- `MQTTMessage`: LogicChan
- `StoreTask`: ChangeChan, CycleChan
- `AlarmTask`: AlarmChan
- `SSEMessage`: SSEChan

## 🧠 全局管理器 (core/)

### `tag_manager.go` - 测点管理器 ⭐⭐⭐
**职责**: 维护内存中的所有测点

## 🐛 当前问题定位

### 问题1: 产量计数器假回落

**文件**: `models/tag.go`、`workers/logic_worker.go`  
**原因**: 设备或网关恢复时可能出现 `77 -> 0 -> 77`，若直接存储 `0`，后续 `0 -> 77` 会被误算为新增产量。  
**现状**: 已通过采集层防抖处理。`suspicious_value` 为空时默认不开启；产量点位建议配置 `suspicious_value=0, debounce_threshold=5`。

---

## 🔧 修复建议

### 修复1: 产量点位启用采集层防抖

在设备配置页填写：

- 可疑抖动值：`0`
- 起滤阈值：`5`
- 启动快照：留空表示兼容旧行为，变化存储/混合存储默认开启

或执行 SQL：

```sql
UPDATE sys_variables
SET suspicious_value = 0,
    debounce_threshold = 5,
    startup_snapshot_enable = NULL
WHERE var_name LIKE '%产量%';
```

详细规则见 `docs/采集层防抖与启动快照说明.md`。

## 📊 数据流向总结

```
MQTT消息
    ↓
LogicChan (2000) → LogicWorker x20 → 更新内存Map
    ↓                                      ↓
    ├─→ ChangeChan (500) → ChangeWorker x5 → 数据库
    ├─→ CycleChan (500)  → CycleWorker x5  → 数据库
    ├─→ AlarmChan (200)  → AlarmWorker x3  → 数据库
    ├─→ EventChan (300)  → EventProcessor x3 → 执行动作
    └─→ SSEChan (100)    → SSE Broadcaster → 前端

Scanner x1 (每1秒) → 扫描内存Map → CycleChan
```
