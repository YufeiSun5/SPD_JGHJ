# 任务去重机制说明

## 🎯 问题背景

在多协程处理架构中，条件任务可能被多个协程同时求值，导致**重复触发**：

```
场景: 条件任务 "a > 1 && b < 5"

协程1: 处理条件事件 → 求值 true → 触发任务
协程2: 处理条件事件 → 求值 true → 触发任务  ❌ 重复！
协程3: 处理条件事件 → 求值 true → 触发任务  ❌ 重复！

结果: 同一个任务被触发 3 次
```

---

## ✅ 解决方案

### **去重机制设计**

1. **仅对条件任务生效**
   - 定时任务：不受影响（单协程扫描）
   - 数据改变任务：不受影响（按变量ID分发，天然隔离）
   - **条件任务**：启用去重（多协程可能重复触发）

2. **去重粒度：100ms**
   - 同一任务在 100ms 内只触发一次
   - 去重键格式：`taskID_timestamp` (timestamp 为 100ms 粒度)
   - 例如：`"123_16800000000"` 表示任务123在某个100ms时间窗口

3. **自动清理**
   - 每 10 秒清理一次过期记录
   - 保留最近 5 秒的记录
   - 防止内存无限增长

## 📊 实现细节

### **1. 数据结构**

```go
type TaskScheduler struct {
    // ... 现有字段
    
    // 🔥 去重机制: 防止条件任务在短时间内被多个协程重复触发
    // key: "taskID_timestamp_ms", value: time.Time
    recentTriggers sync.Map
}
```

### **2. 触发逻辑**

```go
func (ts *TaskScheduler) triggerTask(task *models.Task, triggerTime time.Time, ...) {
    // 🔥 去重逻辑: 仅对条件任务生效
    if task.TaskType == models.TaskTypeCondition {
        dedupeKey := fmt.Sprintf("%d_%d", task.TaskID, triggerTime.UnixMilli()/100)
        
        if _, exists := ts.recentTriggers.LoadOrStore(dedupeKey, triggerTime); exists {
            log.Printf("[TaskScheduler] ⚠️ 条件任务重复触发已拦截: %s", task.TaskName)
            return  // 拦截重复触发
        }
    }
    
    // ... 正常触发逻辑
}
```

### **3. 清理协程**

```go
func (ts *TaskScheduler) runDedupeCleanup() {
    ticker := time.NewTicker(10 * time.Second)
    
    for {
        select {
        case <-ticker.C:
            cutoff := time.Now().Add(-5 * time.Second)
            cutoffMs := cutoff.UnixMilli() / 100
            
            ts.recentTriggers.Range(func(key, value interface{}) bool {
                var taskID, timestamp int64
                fmt.Sscanf(key.(string), "%d_%d", &taskID, &timestamp)
                
                if timestamp < cutoffMs {
                    ts.recentTriggers.Delete(key)  // 删除过期记录
                }
                return true
            })
        }
    }
}
```

## 🧪 测试场景

### **场景1：条件任务去重**

```
时刻 T0: a=0.5, b=6
时刻 T1: a=2, b=3 (满足条件 "a>1 && b<5")

LogicWorker → TriggerChan ← {ConditionData: {a:2, b:3}}

协程1 处理:
  - 任务1 "a>1 && b<5" → true → 触发 ✅
  - 去重记录: "1_16800000000" = T1

协程2 处理 (同时):
  - 任务1 "a>1 && b<5" → true → 检查去重
  - 发现 "1_16800000000" 已存在 → 拦截 ✅

结果: 任务1 只触发 1 次 ✅
```

### **场景2：数据改变任务不受影响**

```
时刻 T0: a=0.5
时刻 T1: a=2 (触发数据改变任务)

LogicWorker → TriggerChan ← {VarID:1, old:0.5, new:2}

协程1 处理:
  - 任务2 (绑定变量a, 类型=数据改变) → 触发 ✅
  - 不检查去重 (TaskType != Condition)

结果: 数据改变任务正常触发 ✅
```

### **场景3：定时任务不受影响**

```
每 60 秒触发一次

定时扫描协程:
  - 任务3 (定时任务) → 触发 ✅
  - 不检查去重 (TaskType != Condition)

结果: 定时任务正常触发 ✅
```

## 📈 性能影响

### **内存占用**

假设：
- 100 个条件任务
- 每秒触发 10 次
- 保留 5 秒记录

```
内存占用 = 100 任务 × 10 次/秒 × 5 秒 × 50 字节/记录
         = 250 KB

✅ 可忽略不计
```

### **CPU 开销**

- `LoadOrStore` 操作：O(1)，极快
- 清理操作：每 10 秒一次，开销很小
- **对性能几乎无影响** ✅

## 🎯 总结

| 任务类型 | 是否去重 | 原因 |
|---------|---------|------|
| **条件任务** | ✅ 是 | 多协程可能重复触发 |
| **数据改变任务** | ❌ 否 | 按变量分发，天然隔离 |
| **定时任务** | ❌ 否 | 单协程扫描，不会重复 |

**去重机制只对条件任务生效，不影响其他模块！** 🎉
