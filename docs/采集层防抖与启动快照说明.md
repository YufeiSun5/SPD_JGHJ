# 采集层防抖与启动快照说明

## 目的

本机制用于处理焊机产量累计计数器在通讯恢复、网关重连、设备重启时出现的假回落值。典型问题是历史计数已经到 `77`，采集链路短暂读到 `0`，下一帧又恢复 `77`。如果直接把 `0` 写入历史表或任务链路，后续 `0 -> 77` 会被误判成新增 77 个良品。

采集层防抖运行在 `workers/logic_worker.go` 更新内存 Tag 之前。假值不会进入内存、任务触发、历史入库、报警或 SSE。

## 与工单数量口径的关系

产量累计计数器的增量代表总产量，应写入或用于修复 `pro_orders.actual_qty`。它不是 OK 数量。

NG 数量来自 NG+1 / NG-1 脉冲净值。现场确认 NG 与 NG-1 已经正确时，历史修复不要覆盖 `ng_qty`。

OK 数量固定按 `ok_qty = actual_qty - ng_qty` 计算。历史修复推荐分两步执行：先按累计产量点位和工单/运行时间窗口修 `actual_qty`，再按公式修 `ok_qty`。

CN: 采集层防抖解决的是总产量累计计数器的假回落，不能把该增量直接当作良品数。  
EN: Acquisition debounce cleans false drops in the cumulative total counter; that delta must not be treated directly as OK quantity.  
JP: 収集層のデバウンスは累積総数カウンタの偽低下を処理するものであり、その増分をそのまま良品数として扱ってはいけません。

## 配置字段

`sys_variables` 增加三列，应用启动时会由 `database.EnsureVariableAcquisitionConfigColumns()` 幂等补齐；也可以手工执行 `database/migrations/add_variable_acquisition_debounce.sql`。

| 字段 | 默认 | 含义 |
| --- | --- | --- |
| `suspicious_value` | `NULL` | 可疑抖动值。`NULL` 表示关闭防抖，产量点位通常填 `0`。 |
| `debounce_threshold` | `5` | 起滤阈值。只有 LastValidValue 大于该值时才拦截可疑值。 |
| `startup_snapshot_enable` | `NULL` | 启动首帧快照开关。`NULL` 表示兼容旧行为，`1` 开启，`0` 关闭。 |

前端入口：设备配置页 `DeviceConfig.vue` 的「可疑抖动值」「起滤阈值」「启动快照」三列。

## 判定规则

### 普通运行中

- 未配置 `suspicious_value`：防抖关闭，按原采集逻辑处理。
- 收到可疑值，且 `LastValidValue > debounce_threshold`：进入内存观察窗，不入库、不触发任务、不更新当前值。
- 下一帧非可疑值 `next >= LastValidValue`：判定中间可疑值是假回落，丢弃可疑值，只处理 `LastValidValue -> next`。
- 下一帧非可疑值 `next < LastValidValue`：判定为真实复位，先补发 `LastValidValue -> suspicious_value`，再处理 `suspicious_value -> next`。
- `LastValidValue <= debounce_threshold`：可疑值直接放行，保护 `1,0,1` 和 `5,0,3` 这类真实小数值复位。
- 观察窗不超时：如果一直停在可疑值，暂不落库，直到下一帧非可疑值到来。

### 启动首帧

开启防抖的点位会在加载时预热 `sys_data_history` 最近非零值，作为启动前 LastValidValue。

- 启动首帧是可疑值，且历史 LastValidValue 大于阈值：先挂起，不执行启动快照 `InitOnly`。
- 下一帧 `next >= 历史 LastValidValue`：丢弃启动可疑值，把 `next` 作为启动快照候选值。
- 下一帧 `next < 历史 LastValidValue`：判定真实复位，补发可疑值，再处理 `next`。
- 找不到历史非零值：保持兼容逻辑，不强行挂起首帧。

## 示例

| 序列 | 阈值 | 结果 |
| --- | ---: | --- |
| `10,0,12` | `5` | 丢弃 `0`，处理 `10 -> 12` |
| `10,0,1` | `5` | 补发 `10 -> 0`，再处理 `0 -> 1` |
| `10,0,0,11` | `5` | 丢弃中间所有 `0` |
| `10,0,0,1` | `5` | 补发一个 `0`，再处理 `1` |
| `1,0,1` | `5` | `0` 直接放行 |
| `5,0,3` | `5` | `0` 直接放行 |
| `6,0,7` | `5` | `0` 挂起后丢弃 |
| 启动历史 `77`，首帧 `0`，下一帧 `77` | `5` | 启动 `0` 不入库，`77` 作为首帧快照候选 |

## 验证路径

后端验证：

```powershell
go test ./...
go build ./...
```

前端验证需要使用能找到 vfox Node/npm 的 PowerShell 7 环境：

```powershell
cd desktop/frontend
npm run build
```

数据库复查：

```sql
SHOW COLUMNS FROM sys_variables
WHERE Field IN ('suspicious_value', 'debounce_threshold', 'startup_snapshot_enable');

SELECT id, var_name, suspicious_value, debounce_threshold, startup_snapshot_enable
FROM sys_variables
WHERE suspicious_value IS NOT NULL;
```

页面验证：

1. `desktop` 目录运行 `wails dev`。
2. 打开设备配置页。
3. 产量累计计数器点位配置 `suspicious_value=0`、`debounce_threshold=5`。
4. `startup_snapshot_enable` 留空表示兼容旧行为，也就是变化存储/混合存储默认开启首帧快照。
