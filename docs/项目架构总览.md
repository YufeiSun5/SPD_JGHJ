项目背景与架构全景： 我们正在开发一个基于Golang、VUE3、MQTT、Wails的高性能工业物联网（IIoT）网关系统。

核心设计理念：

内存为王 (Memory First)：所有测点（Tag）的配置（上下限、存储模式）和实时值都维护在内存的 map[string]*Tag 中，使用 sync.RWMutex 保证并发读写安全。

计算与 IO 分离：MQTT 接收和逻辑判断（计算密集）在高速通道；存库和报警（IO 密集）在低速通道，通过 Buffered Channel 解耦。

无 Redis 架构：利用 Go 原生 Channel 和 Map 替代 Redis 的缓存和消息分发功能。

真并行架构 (True Parallelism)：🔥 Go 的协程（goroutine）是真正的操作系统线程级并行，不像 Python 的 GIL（全局解释器锁）限制。系统在多核 CPU 上可以同时运行多个协程，实现真正的并发处理。例如：20个 LogicWorker 可以在 20 个 CPU 核心上同时处理 MQTT 消息，5个 TaskScheduler 处理协程可以并行求值条件表达式，3个 EventProcessor 可以同时执行 HTTP 请求、MQTT 发布和数据库操作。Go 的 runtime 调度器会自动将协程分配到可用的 CPU 核心上，充分利用硬件资源。

双循环一致性 (PLC Scan Cycle)：模仿 PLC 扫描周期，Pass 1 建立输入快照，Pass 2 基于快照执行业务逻辑，确保脚本/规则引擎读取的变量是同一时刻的值，避免时空错乱。这个改进了北京亚控KingSCADA的逐变量更新模式，比如焊接监控场景：当"焊接电流>=200A 且 焊接电压>=25V"时触发合格判定脚本，如果上一周期电流=180A、电压=22V，本周期实际数据是电流=210A、电压=28V。在KS的逐变量更新模式下，先更新电流=210A时（电压仍是22V），条件不满足；等更新电压=28V时，电流可能已经跳变为190A，条件依然不满足，导致本应判定为合格的焊点被误判为不合格。类似地，在锅炉安全监控中，"蒸汽压力>8MPa 且 水位<30%"应触发紧急停炉，但逐变量更新可能错过这个危险瞬间。我设计的基于快照更新数据的方式参考了WinCC和西门子软件的数据流设计，确保所有相关变量在同一时刻采样，彻底解决了工业控制中的时序一致性问题。

并发架构详解 (Concurrency Architecture)：

单进程多协程模型 (Single Process, Multi-Goroutine)：
  - 🔥 所有协程运行在同一个 Go 进程中
  - 🔥 Go runtime 自动将协程调度到多个 CPU 核心
  - 🔥 真正的并行执行（不像 Python GIL 限制为单核）
  - 🔥 协程间通过 Channel 通信，零拷贝，极速
  - 🔥 共享内存访问（TagManager、任务列表等）使用读写锁保护

实际性能表现（8核CPU）：
  - LogicWorker (20个): 可同时在 8 个核心上并行处理 MQTT 消息
  - TaskScheduler (5个): 可同时在 5 个核心上并行求值条件表达式
  - EventProcessor (3个): 可同时在 3 个核心上并行执行 HTTP/MQTT/DB 操作
  - 总吞吐量: 单机可处理 10,000+ 消息/秒 (8核CPU)

关键协程设计 (The 10-Worker Model + Task System)： 系统启动时需初始化以下协程池，各司其职：

1. MQTT 监听器 (Paho Internal, 每网关1个)：
   - 负责网络收发与自动重连
   - 绝对不能阻塞
   - 多网关独立运行

2. 核心逻辑池 (Logic Processors, x20)：
   - 消费 LogicChan (缓冲2000)
   - 双循环快照模式 (Pass 1: 建立快照, Pass 2: 执行业务)
   - 加锁更新内存 Map
   - 分发任务到下游 Channel
   - ⚠️ 非阻塞：所有下游操作通过 Channel 异步

3. 变动存储工 (Change Workers, x5)：
   - 消费 ChangeChan (缓冲500)
   - 批量写入优化：每100ms或累积10条
   - 处理 store_mode=1/3 的变动存储

4. 定时存储工 (Cycle Workers, x5)：
   - 消费 CycleChan (缓冲500)
   - 批量写入优化：每200ms或累积20条
   - 处理 store_mode=2/3 的定时存储

5. 定时扫描器 (Scanner, x1)：
   - 单例协程，每1秒扫描内存 Map
   - 检查定时存储周期，发送到 CycleChan
   - 非阻塞发送

6. 报警执行工 (Alarm Workers, x3)：
   - 消费 AlarmChan (缓冲200)
   - 写报警日志、更新报警状态
   - 可扩展：发邮件/短信
   - ⚠️ 极慢IO，独立协程池

7. SSE 广播员 (SSE Broadcaster, x1)：
   - 消费 SSEChan (缓冲100)
   - 维护前端长连接
   - 非阻塞广播

8. 任务调度器 (Task Scheduler, x7)：
   - 1个定时扫描协程：每1秒检查定时任务
   - 5个触发处理协程：并发处理数据改变/条件任务
   - 1个去重清理协程：每10秒清理过期去重记录
   - LogicWorker异步触发 → TriggerChan (缓冲10000)
   - 条件事件任务求值（已实现表达式引擎 expr-lang/expr）
   - 🔥 不丢弃任务：超大缓冲队列，队列满时阻塞等待
   - 🔥 去重机制：条件任务100ms内只触发一次，防止多协程重复触发
   - 非阻塞发送到 EventChan

9. 事件处理器池 (Event Processors, x3)：
   - 消费 EventChan (缓冲300)
   - 执行5种动作：HTTP/MQTT/DB/Script/Log
   - MQTT发布使用触发消息的网关 ✅
   - 数据库写入直接执行（单条，非批量）✅
   - 异步保存任务执行日志 ✅

10. 多网关管理器 (Gateway Manager)：
    - 管理多个MQTT网关连接
    - 每个网关独立协程
    - 支持热重载（增删改网关）

数据交互流程 (Pipeline)：

【数据存储流程】

Ingress：MQTT 收到消息 -> 放入 LogicChan (缓冲2000)。

Process：LogicWorker 双循环处理：
  - Pass 1: 建立快照 -> 获取 RWMutex 写锁 -> 更新内存 Value -> 释放锁 -> 记录变化
  - Pass 2: 基于快照执行业务逻辑（此时所有变量已更新完毕）

Dispatch（非阻塞发送）：
  - 如果值变动 & store_mode=1/3 -> 放入 ChangeChan (缓冲500)
  - 如果触发报警规则 -> 放入 AlarmChan (缓冲200)
  - 如果需要前端展示 -> 放入 SSEChan (缓冲100)
  - 如果值改变 & 有绑定任务 -> 触发 TaskScheduler

Egress：
  - ChangeWorker x5: 批量写入（100ms/10条）
  - CycleWorker x5: 批量写入（200ms/20条）
  - AlarmWorker x3: 单条写入 + 通知

【任务系统流程】✅ 已完整实现

Scheduler：
  - TaskScheduler 定时扫描协程 (x1)：每1秒检查定时任务 -> 触发 -> 放入 EventChan

Trigger：
  - LogicWorker Pass 2 检测数据改变
  - 调用 TriggerDataChangeTask(varID, old, new, timestamp, gatewayID)
  - 🔥 异步发送到 TriggerChan (缓冲10000，不丢弃)
  - TaskScheduler 触发处理池 (x5)：并发消费 TriggerChan
  - 支持 FALSE_TO_TRUE/TRUE_TO_FALSE/ANY/INCREASE/DECREASE/THRESHOLD
  - 条件表达式求值：支持 "temp>50 && pressure>100" 等复杂逻辑
  - 非阻塞放入 EventChan (缓冲300)

Execute：
  - EventProcessor x3 并发消费 EventChan
  - 执行动作：HTTP/MQTT/DB/Script/Log
  - MQTT发布使用触发消息的网关（网关感知）✅
  - 数据库操作直接执行（非批量）
  - 异步保存执行日志到 sys_task_execution_logs

关键实现细节 (Implementation Details)：

【非阻塞设计】
  - 大部分 Channel 使用 select + default，满了则丢弃并记录日志
  - TaskScheduler -> EventChan: 非阻塞发送 ✅
  - LogicWorker -> ChangeChan/AlarmChan/SSEChan: 非阻塞发送 ✅
  - 🔥 LogicWorker -> TriggerChan: 阻塞发送，但缓冲10000，不丢弃任务 ✅
  - 任务执行日志: 异步 goroutine 保存，不阻塞主流程 ✅

【批量写入优化】
  - ChangeWorker: 每100ms 或累积10条批量写入
  - CycleWorker: 每200ms 或累积20条批量写入
  - AlarmWorker: 单条写入（报警数量少）
  - EventProcessor 数据库操作: 单条直接执行（任务触发频率低）✅

【网关感知 MQTT 发布】✅ 新增
  - LogicWorker 触发任务时携带 gatewayID
  - EventProcessor 优先使用触发消息的网关发布
  - 如果该网关不可用，自动切换到其他活跃网关
  - 支持多网关负载均衡

【配置热重载】
  - 监控 sys_config_version 表版本变化（每10秒检查）
  - 自动重载: 测点配置/网关配置/任务配置 ✅
  - 无需重启系统

【双循环一致性保证】
  - Pass 1 完成后，内存中所有变量已更新到 t 时刻
  - Pass 2 执行任务时，脚本读取任何变量都是 t 时刻的值
  - 避免时空错乱：温度读的是新值，压力读的是旧值

【冷启动保护机制】✅ 新增
  - 问题: 系统启动时 Tag 初始化为 0，第一条 MQTT 消息会产生错误的 change 值
  - 场景: 设备计数器实际值=1000，系统重启后收到第一条消息，会误判为 +1000
  - 解决: Tag.IsFirstUpdate 标志，首次更新时只建立基准值，不触发任务
  - 实现: models/tag.go UpdateValue()/UpdateStringValue() 检查 IsFirstUpdate
  - 效果: 系统启动后第一条消息不触发任务，从第二条开始正常计算增量

【存储模式详解 store_mode】
  0 - 不存储: 不写入数据库，用于临时计算变量
  
  1 - 变动存储: 值变化超过 store_deadband 时存储
      优点: 节省存储空间，只记录变化
      缺点: 值长期不变时无记录，图表可能断线
      适用: 开关状态、阀门开度等离散量
  
  2 - 定时存储: 每隔 store_cycle 秒存储一次
      优点: 数据连续性好，周期可预测
      缺点: 快速变化过程可能丢失，数据失真
      适用: 缓慢变化的环境参数（湿度、环境温度）
  
  3 - 混合存储 ⭐ 推荐: 变动存储 + 定时存储双保险
      触发条件: 变化超过死区 OR 周期到期
      优点: 既不漏快速变化，又保证稳定期有记录
      适用: 90%工业测点（温度、压力、流量、计数器）
      示例: store_mode=3, store_cycle=60, store_deadband=1.0
            → 变化>1.0立即存 或 60秒必存一次

【触发阶段 vs 执行阶段】分离设计

触发阶段 (TaskScheduler - 7个协程)：
  ┌────────────────────────────────────────────────────────┐
  │ 协程1: 定时任务扫描器                                   │
  │   - 每1秒检查 scheduledTasks                           │
  │   - 判断 LastRunTime + IntervalSec                     │
  │   - 直接调用 triggerTask() 发送到 EventChan          │
  ├────────────────────────────────────────────────────────┤
  │ 协程2-6: 触发事件处理池 (5个并发)                      │
  │   - 消费 TriggerChan (缓冲10000)                       │
  │   - 处理3种触发事件:                                   │
  │     • data_change: 遍历 dataChangeTasks               │
  │       检查 change_type (ANY/INCREASE/DECREASE/...)    │
  │     • data_change_string: 遍历字符串任务              │
  │     • condition: 遍历 conditionTasks                  │
  │       调用 evaluateCondition() 求值表达式             │
  │   - 🔥 去重检查: 条件任务100ms内只触发一次            │
  │   - 调用 triggerTask() 发送到 EventChan              │
  ├────────────────────────────────────────────────────────┤
  │ 协程7: 去重记录清理器                                  │
  │   - 每10秒清理一次 recentTriggers (sync.Map)          │
  │   - 删除超过5秒的去重记录                             │
  └────────────────────────────────────────────────────────┘
  
  特点: 
    ✅ 快速 (<1ms): 只做条件判断和表达式求值
    ✅ 并行: 5个协程可同时处理不同变量的触发事件
    ✅ 不阻塞: 发送到 EventChan 后立即返回
    ✅ 线程安全: 使用 RLock 读取任务列表，允许多协程并发

执行阶段 (EventProcessor - 3个协程)：
  ┌────────────────────────────────────────────────────────┐
  │ 协程1-3: 事件处理器 (3个并发)                          │
  │   - 消费 EventChan (缓冲300)                           │
  │   - 执行5种动作:                                       │
  │     1. HTTP 请求: 发送 GET/POST/PUT                   │
  │     2. MQTT 发布: 通过网关发布消息                    │
  │     3. 数据库操作: 执行 SQL 或预定义操作              │
  │     4. 脚本执行: 运行 bash/python/powershell          │
  │     5. 日志写入: 输出到控制台/文件                    │
  │   - 模板变量替换: {{var_id}} → 实际值                │
  │   - 异步保存执行日志 (不阻塞)                         │
  └────────────────────────────────────────────────────────┘
  
  特点:
    ✅ 慢速 (10-1000ms): 涉及网络IO、数据库IO、脚本执行
    ✅ 并行: 3个协程可同时执行不同的任务动作
    ✅ 隔离: 执行慢不影响触发快速完成
    ✅ 可靠: 执行日志异步保存，失败不影响主流程

  ↓

【并发安全保证】

1. 任务列表读取 (TaskScheduler):
   - 使用 RWMutex.RLock() 允许多协程并发读取
   - 只有 LoadTasks() 时才加写锁
   - 5个触发处理协程可同时读取任务列表

2. 去重机制 (sync.Map):
   - LoadOrStore() 是原子操作，线程安全
   - 多个协程同时触发同一任务，只有第一个成功
   - 清理协程定期删除过期记录，不影响触发

3. 内存变量访问 (TagManager):
   - 使用 RWMutex 保护 map[string]*Tag
   - LogicWorker 写入时加写锁
   - 其他协程读取时加读锁

【性能优势】

对比单协程设计:
  - 触发处理: 1个协程 → 5个协程 = 5倍吞吐量
  - 任务执行: 1个协程 → 3个协程 = 3倍吞吐量
  - 总体提升: 约 5-10 倍处理能力

对比 Python (GIL 限制):
  - Python: 多线程受 GIL 限制，实际单核运行
  - Go: 真并行，45个协程可分布在多个CPU核心
  - 性能差距: 8核CPU上约 8-10 倍性能优势

实际场景 (8核CPU, 1000个变量, 每个绑定10个任务):
  - 触发阶段: 5个协程并行匹配，每秒可处理 5000+ 触发事件
  - 执行阶段: 3个协程并行执行，每秒可完成 300+ 任务动作
  - 队列缓冲: TriggerChan (10000) + EventChan (300) 应对突发流量

【为什么分离触发和执行？】

1. 解耦: 触发逻辑不会被慢速IO阻塞
2. 并发: 可以独立调整触发和执行的协程数量
3. 缓冲: EventChan 缓冲300，应对执行慢于触发的情况
4. 可靠: 触发快速完成，执行慢慢处理，不会丢失任务
5. 监控: 可以分别监控触发队列和执行队列的堆积情况
