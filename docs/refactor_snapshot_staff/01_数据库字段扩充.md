## 1. 新增字段

在 `ProShiftSnapshot` 结构体中新增 **1 个字段**：

### `sessions_snapshot` — 多 session 工时明细 JSON

```go
// SessionsSnapshot 存放该班次时间窗口内所有关联 session 的工时拆分明细。
// CN: 当同一班次有多段上机记录（跨班交接/中途换人）时，每段都独立记录重叠时长和占比。
// EN: When a shift window overlaps multiple machine sessions (handover / mid-shift swap),
//     each session's overlap duration and percentage are stored independently.
// JP: 同一シフト時間帯に複数の上機セッション（交替・途中交代）がある場合、
//     各セッションの重複秒数と割合を独立記録する。
SessionsSnapshot string `gorm:"type:json;comment:多session工时拆分明细JSON" json:"sessions_snapshot"`
```

**JSON 结构定义**：

```go
// SessionSnapshotItem 单条 session 在该班次时间窗口内的工时快照
type SessionSnapshotItem struct {
    SessionID  int64  `json:"session_id"`   // pro_machine_sessions.id
    TeamID     int    `json:"team_id"`      // 该 session 的班组 ID
    TeamName   string `json:"team_name"`    // 班组名称
    StaffIDs   []int  `json:"staff_ids"`    // 该 session 的当班人员 ID 列表
    LoginTime  string `json:"login_time"`   // 原始登录时间 "HH:MM"
    LogoutTime string `json:"logout_time"`  // 原始登出时间 "HH:MM"（进行中为空）
    OverlapSec int    `json:"overlap_sec"`  // 与班次时间窗口的有效重叠秒数
    Pct        float64 `json:"pct"`         // 占班次总工时的百分比
}
```

## 2. 扩充现有字段内容

### `staff_snapshot` — 原有 JSON 加入工时信息

**改造前格式**：
```json
[{"id":5,"name":"王五","code":"W005"}]
```

**改造后格式**（向后兼容，新增 `team_id`、`team_name`、`work_sec`、`pct` 字段）：
```json
[
  {"id":3,"name":"张三","code":"Z003","team_id":1,"team_name":"A组","work_sec":1800,"pct":6.25},
  {"id":4,"name":"李四","code":"L004","team_id":2,"team_name":"B组","work_sec":23400,"pct":81.25}
]
```

对应的 Go 结构体修改：

```go
// StaffSnapshotItem 人员快照条目（含工时拆分）
type StaffSnapshotItem struct {
    ID       int     `json:"id"`
    Name     string  `json:"name"`
    Code     string  `json:"code"`
    TeamID   int     `json:"team_id,omitempty"`   // 所属 session 的班组 ID
    TeamName string  `json:"team_name,omitempty"` // 班组名称
    WorkSec  int     `json:"work_sec,omitempty"`  // 该员工在本班次的有效工作秒数
    Pct      float64 `json:"pct,omitempty"`       // 工时占比(%)
}
```

> **兼容性说明**：旧快照的 `staff_snapshot` 只有 `id/name/code`，JSON 反序列化时新增字段自动为零值，前端判断 `work_sec > 0` 即可区分新旧数据。

## 3. 保留字段的行为变更

| 字段 | 改造前 | 改造后 |
|------|--------|--------|
| `team_id` | 最后一条 session 的 team_id | **占比最大**的 session 的 team_id（主班组） |
| `team_name` | 最后一条 session 的 team_name | **占比最大**的 session 的 team_name |
| `session_id` | 最后一条 session 的 ID | **占比最大**的 session 的 ID |
| `staff_snapshot` | 仅最后一条 session 的人员 | **所有** session 的人员（含工时） |

> **为什么保留 `team_id`/`team_name`/`session_id`**？  
> - 这三个字段被 `GetShiftSnapshots` 的 `WHERE team_id = ?` 筛选条件使用  
> - 它们代表"该班次的主责班组"概念，对现有报表兼容且直观  
> - 详细的多 session 数据走 `sessions_snapshot` JSON 字段
